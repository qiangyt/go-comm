|---------|-------|------|

## 性能指标说明

| 指标 | 计算公式 | 示例 |
|------|---------|------|
| **提升** | `(原值 - 新值) / 原值 × 100%` | 912.5 → 320.7 ns/op = **64.9%** ↓ |
| **速度倍增** | `原值 / 新值` | 912.5 / 320.7 = **2.85x** |

> **提升 64.9%** = 时间减少 64.9% = 速度提升 **2.85 倍**

## 配置切换 API

```go
import "github.com/qiangyt/go-comm/v2"

// 切换到标准库
comm.JSONConfig.Backend = comm.JSONBackendStdlib

// 切换到 sonic（默认）
comm.JSONConfig.Backend = comm.JSONBackendSonic
```

## 总结

### 主要发现

1. **单线程 Unmarshal 性能提升显著**:
   - 31 bytes: **64.9%** 提升, **2.85x** 加速 (912.5 → 320.7 ns/op)
   - 128 bytes: **55.0%** 提升, **2.22x** 加速 (3503 → 1578 ns/op)
   - ~5.5KB: **45.3%** 提升, **1.83x** 加速 (82446 → 45140 ns/op)

2. **单线程 Marshal 性能提升显著**:
   - 31 bytes: **43.8%** 提升, **1.78x** 加速 (540.0 → 303.5 ns/op)
   - 128 bytes: **46.5%** 提升, **1.87x** 加速 (1727 → 923.8 ns/op)

3. **并发性能提升更加明显**:
   - Unmarshal 并发: **62.6% ~ 73.2%** 提升, **2.67x ~ 3.73x** 加速
   - Marshal 并发: **63.6% ~ 72.8%** 提升, **2.75x ~ 3.67x** 加速

4. **内存分配减少**:
   - FromJson: 内存减少 **56.8%**，分配次数减少 **58.3%**
   - ParseCommandOutput: 内存减少 **14.5%**，分配次数减少 **30.8%**

5. **动态配置切换**:
   - go-comm 提供全局配置变量 `JSONConfig.Backend`
   - 可在运行时动态切换 stdlib 和 sonic
   - 方便进行性能对比和兼容性测试

### 分析

- sonic 通过 SIMD 指令和 JIT 编译实现了显著的性能提升
- 在 QEMU 虚拟环境中，sonic 的汇编优化可能无法完全发挥，实际物理机上性能差异可能更大
- 内存分配的减少是实实在在的优化效果，有助于降低 GC 压力
- 并发场景下 sonic 的优势更加明显，适合高并发应用
